## 4. ML 모델 재학습 계획

### 4.1 재학습 기준

| 구분 | 기준 | 조치 |
|------|------|------|
| 정기 재학습 | 분기별 1회 | 신규 데이터 추가 후 전체 재학습 |
| 성능 저하 감지 | RMSE 증가 20% 이상 | 즉시 재학습 검토 |
| 목표값 변경 | SN/AG/CU 목표값 변경 시 | 피처 재생성 후 재학습 |
| 신규 공급사 추가 | 새 공급사 데이터 축적 후 | 재학습 권장 |

### 4.2 재학습 절차

```
① 학습 데이터 추출
   - data/raw/formulation_history.csv 업데이트
   - 신규 LOT 데이터 포함 (최소 200건 이상 추가 시 효과적)

② 데이터 품질 검증
   - 성분 합계 100% 검증
   - 이상값(outlier) 제거
   - 결측값 처리

③ 모델 학습 실행
   python scripts/train.py --data formulation_history.csv \
     --target quality_score --model gradient_boosting

④ 모델 성능 비교
   - 이전 모델 vs 신규 모델 RMSE/R² 비교
   - 신규 모델이 개선되었을 때만 교체

⑤ 모델 배포
   - models/artifacts/gradient_boosting.joblib 교체
   - FastAPI 서버 재시작

⑥ 재학습 결과 기록
   - ML 모델 이력 DB 업데이트
   - 월간 KPI 보고서 반영
```

## 5. 장애 대응 절차

### 5.1 장애 등급 분류

| 등급 | 기준 | 예시 | 대응 시간 |
|------|------|------|-----------|
| P1 (긴급) | 서비스 전면 중단 | 서버 다운, DB 연결 불가 | 2시간 이내 |
| P2 (심각) | 핵심 기능 장애 | ML API 응답 불가, 로그인 불가 | 4시간 이내 |
| P3 (보통) | 일부 기능 이상 | 특정 화면 오류, 느린 응답 | 업무일 이내 |
| P4 (경미) | 사용성 불편 | UI 표시 오류, 문구 수정 | 다음 패치 반영 |

### 5.2 P1 장애 대응 절차

```
[장애 감지]
  사용자 신고 or 모니터링 알림
  │
  ▼ (5분 이내)
[담당자 확인]
  이성민 (시스템 담당) → 현상 파악
  │
  ▼ (30분 이내)
[원인 분석]
  docker logs, nginx error.log, PostgreSQL log 확인
  │
  ▼ (1시간 이내)
[긴급 조치]
  서비스 재시작 또는 롤백
  │
  ▼ (2시간 이내)
[서비스 복구 확인]
  헬스체크 API 확인
  사용자 접속 확인
  │
  ▼
[사후 보고]
  장애 원인, 조치 내용, 재발 방지 계획 문서화
```

## 6. 변경 관리

### 6.1 변경 요청 프로세스

```
[사용자/도입기업 변경 요청]
  이메일 또는 티켓 등록
  │
  ▼
[변경 요청 검토]
  박준혁 (기획) → 범위/영향도 분석
  │
  ▼
[변경 승인]
  김현수 (운영 총괄) 승인
  │
  ▼
[개발/배포]
  이성민/장다운 → 개발, 테스트, 배포
  │
  ▼
[변경 완료 확인]

```
