# SF-OS1 운영지원계획서

**문서번호**: SF-OS1 | **버전**: V1.0 | **작성일**: 2026-06-19 | **작성자**: 기획팀

---

## 1. 문서 개요

### 1.1 목적

본 문서는 Formulation ML 시스템(성분분석 데이터 기반 배합비율 최적화 ML 시스템)의 안정적인 운영을 위한 조직 체계, 운영 절차, 유지보수 계획, 교육 계획 및 서비스 수준 협약(SLA)을 정의한다.

시스템 도입 후 지속적인 운영 품질을 확보하고, 장애 발생 시 신속한 대응 및 복구가 가능하도록 체계적인 운영 지원 방안을 수립한다.

### 1.2 적용 범위

- 대상 시스템: Formulation ML (FastAPI Backend + Next.js Frontend)
- 운영 환경: 사내 인프라 (Backend port 8000, Frontend port 3000)
- 적용 조직: 스마트팀(IT), 생산팀, 품질팀, 기술팀

### 1.3 참조 문서

- SF-TD1 개선업무설계서
- SF-DS1 시스템설계서
- SF-OS2 운영현황보고서

## 2. 운영 조직 체계

### 2.1 운영 조직도

```
[운영 책임자]
스마트팀장
    │
    ├── [시스템 운영] 스마트팀 / IT팀
    │       ├── 시스템 관리자 (서버·배포·장애 대응)
    │       └── 데이터 관리자 (학습데이터·모델 재학습)
    │
    ├── [업무 운영] 생산팀 / 기술팀
    │       ├── 공정 엔지니어 (DOE 설계·공정 최적화)
    │       └── 공정 운영자 (일상 배합 추천 활용)
    │
    └── [품질 관리] 품질팀
            └── 품질관리 담당자 (품질 판정·모델 성능 모니터링)
```

**외부 지원 체계**:
- 개발팀(내부 또는 외부): 기능 개선, 버그 수정, 신규 모델 개발 지원
- 인프라팀: 서버 하드웨어, 네트워크 지원

### 2.2 역할 및 책임

| 역할 | 소속 | 주요 책임 | 비상 연락 |
|------|------|-----------|-----------|
| 운영 책임자 | 스마트팀장 | 운영 전략 결정, SLA 이행 감독, 주요 장애 최종 보고 수신 | 우선 연락 |
| 시스템 관리자 | 스마트팀/IT팀 | 서버 기동/중단, 배포 관리, 1·2등급 장애 1차 대응, 정기 점검 실행 | 1차 연락 |

| 공정 운영자 | 생산팀 | 시스템 일상 사용, 공정 데이터 입력, 1차 이상 현상 보고 | 조장 경유 |

## 3. 운영 지원 방안

### 3.1 일상 운영 절차

**시스템 기동 절차 (일과 시작 시)**

1. Backend 서버 상태 확인: `http://localhost:8000/health` 응답 확인
2. Frontend 서버 상태 확인: `http://localhost:3000` 접속 확인
3. 최신 학습 데이터 정합성 확인 (data/raw/formulation_history.csv 파일 존재 및 행 수 확인)
4. 모델 아티팩트 파일 존재 확인 (`models/artifacts/gradient_boosting.joblib`, `preprocessors_gradient_boosting.joblib`)
5. 정상 확인 후 운영자 대기 상태 전환

**일과 중 운영 모니터링**

| 항목 | 확인 주기 | 확인 방법 | 이상 기준 |
|------|----------|-----------|-----------|
| API 응답 시간 | 1시간 | 서버 로그 확인 | 3초 초과 |
| 배합 추천 오류율 | 수시 | 사용자 보고 수신 | 연속 3건 실패 |
| 품질 예측 정확도 | 일 1회 | 실측 vs 예측 비교 | MAPE 10% 초과 |
| 학습 데이터 적재 | 일 1회 | CSV 파일 행 수 확인 | 당일 LOT 누락 |

**시스템 종료 절차 (일과 종료 후 — 해당 시)**

1. 당일 LOT 이력 데이터 저장 완료 확인
2. 처리 중인 DOE 시뮬레이션 작업 완료 대기
3. 서버 정상 종료 (강제 종료 금지)
4. 일일 운영 일지 작성

### 3.2 장애 대응 절차 (1, 2, 3등급 분류)

**장애 등급 정의**

| 등급 | 정의 | 예시 | 목표 복구 시간 |
|------|------|------|---------------|
| 1등급 (Critical) | 시스템 전면 중단 — 모든 사용자 사용 불가 | Backend 서버 다운, Frontend 접속 불가, DB 오류 | 2시간 이내 |
| 2등급 (Major) | 핵심 기능 장애 — 배합 추천 또는 품질 예측 불가 | 모델 로드 실패, API 에러 반복 발생, 예측값 비정상 | 4시간 이내 |
| 3등급 (Minor) | 부가 기능 장애 — 일부 기능 제한 | DOE 시뮬레이터 오류, EDA 차트 미표시, 특정 화면 오류 | 24시간 이내 |

**1등급 장애 대응 절차**

```
장애 감지
    → [즉시] 시스템 관리자에게 연락
    → [10분 이내] 장애 상황 파악 및 영향도 확인
    → [15분 이내] 운영 책임자 보고 (전화 또는 메시지)
    → [30분 이내] 1차 복구 시도 (서버 재기동, 환경 확인)
    → 복구 성공: 정상 운영 확인 후 이슈 등록
    → 복구 실패: 개발팀 긴급 지원 요청 + 수동 운영 전환
    → [2시간 이내] 복구 완료 및 운영 재개 보고
    → 사후 원인 분석 및 재발 방지 계획 수립 (3일 이내)
```

**2등급 장애 대응 절차**

```
장애 감지 (사용자 보고 또는 모니터링)
    → [30분 이내] 시스템 관리자 + 데이터 관리자 공동 확인
    → 모델 파일 오류: 백업 모델로 즉시 교체
    → API 에러: 로그 분석 후 재기동 또는 패치
    → 예측값 이상: 모델 재검증 후 재학습 여부 결정
    → [4시간 이내] 복구 완료
    → 이슈 대장 등록 및 원인 분석 보고 (1주일 이내)
```

**3등급 장애 대응 절차**

```
장애 감지 (사용자 보고)
    → 시스템 관리자 이슈 접수 및 재현 확인
    → 우선순위 평가 (업무 영향도 기준)
    → 개발팀에 버그 수정 요청
    → [24시간 이내] 패치 적용 또는 임시 우회 방법 안내
    → 이슈 대장 등록
```

**수동 운영 전환 절차 (1등급 복구 불가 시)**

- ML 추천 불가 시: 직전 최적 배합비율 이력 조회 → 품질관리 담당자 수동 결정
- 데이터 입력 불가 시: 종이 작업일지 임시 사용 → 시스템 복구 후 일괄 입력

### 3.3 데이터 백업 및 복구 계획

**백업 대상 및 주기**

| 백업 대상 | 백업 주기 | 보관 기간 | 백업 방법 |
|-----------|----------|----------|-----------|
| 학습 데이터 (formulation_history.csv) | 일 1회 | 90일 | 로컬 + 네트워크 드라이브 복사 |
| 모델 아티팩트 (.joblib) | 모델 재학습 전 | 최근 3버전 | `models/artifacts/backup/` 폴더 |
| 전처리기 파일 (preprocessors_*.joblib) | 모델 재학습 전 | 최근 3버전 | `models/artifacts/backup/` 폴더 |
| 시스템 설정 파일 | 주 1회 | 30일 | 네트워크 드라이브 |
| 운영 로그 | 일 1회 | 30일 | 로컬 압축 보관 |

**백업 명명 규칙**:
- 학습 데이터: `formulation_history_YYYYMMDD.csv`
- 모델: `gradient_boosting_YYYYMMDD.joblib`
- 전처리기: `preprocessors_gradient_boosting_YYYYMMDD.joblib`

**복구 절차**

1. 장애 시점 이전 최신 백업 파일 확인
2. 현재 파일을 `_corrupt` 접미사로 이름 변경 (원본 보존)
3. 백업 파일을 운영 경로에 복사
4. 시스템 재기동 후 정상 동작 확인
5. 복구 완료 시간 및 데이터 손실 범위 기록

**목표 복구 시점 (RPO)**: 최대 24시간 전 데이터까지 복구 가능
**목표 복구 시간 (RTO)**: 데이터 복구 후 30분 이내 서비스 재개

### 3.4 모델 재학습 주기 및 절차

**재학습 트리거 조건**

| 조건 | 기준 | 재학습 유형 |
|------|------|------------|
| 정기 재학습 | 분기 1회 (3개월) | 계획 재학습 |
| 데이터 누적 | 신규 LOT 300건 추가 시 | 계획 재학습 |
| 성능 저하 | MAPE > 8% 또는 R² < 0.5 | 긴급 재학습 |
| 불량률 급증 | 주간 불량률 > 5% | 긴급 재학습 |
| 공급사 변경 | 신규 공급사 추가 또는 주요 변경 | 조건부 재학습 |

**재학습 절차**

```
1. 재학습 필요성 확인
   - 성능 지표 확인 (MAE / RMSE / R² / MAPE)
   - 최근 불량 LOT 패턴 분석

2. 데이터 준비
   - formulation_history.csv 최신 데이터 확인
   - 이상치·결측치 점검 및 보완
   - 데이터 건수 확인 (최소 200건 이상 권장)

3. 현행 모델 백업
   - models/artifacts/backup/ 에 날짜 포함 파일명으로 백업
   - 전처리기 파일도 함께 백업

4. 재학습 실행
   python scripts/train.py \
     --data formulation_history.csv \
     --target quality_score \
     --model gradient_boosting

5. 성능 평가 및 비교
   - 신규 모델 vs 현행 모델 성능 지표 비교
   - 신규 모델 성능 개선 확인 (RMSE 감소 또는 R² 증가)

6. 승인 및 교체
   - 데이터 관리자 + 품질관리 담당자 성능 검토 회의
   - 승인 시 운영 모델 교체

7. 변경 이력 기록
   - 재학습일, 학습 데이터 건수, 이전/신규 성능 지표 기록
```

### 4.2 소프트웨어 업데이트 절차

| 유형 | 예시 | 적용 절차 |
|------|------|-----------|

**공통 업데이트 절차**:
1. 업데이트 내용 및 영향도 분석
2. 현행 시스템 전체 백업
3. 스테이징 환경 적용 및 검증
4. 운영 적용 (업무 종료 후 권장)
5. 운영 환경 검증 (주요 기능 동작 확인)
6. 변경 이력 기록

**라이브러리 버전 관리**:
- `requirements.txt` 버전 고정 유지
- 라이브러리 업데이트 시 `requirements.txt` 동시 갱신
- 주요 라이브러리: FastAPI 0.137.1, scikit-learn, XGBoost, scipy

## 5. 사용자 교육 계획

### 5.1 교육 대상 및 커리큘럼

**교육 과정 1: 기본 사용자 교육 (공정 운영자)**

- 대상: 생산팀 공정 운영자
- 교육 시간: 2시간
- 교육 내용:
  1. 시스템 개요 및 도입 목적 (20분)
  2. 시스템 접속 및 화면 구성 안내 (20분)
  3. 배합비율 추천 기능 사용법 — 실습 (30분)
  4. 품질점수 예측 기능 사용법 — 실습 (20분)
  5. LOT 이력 조회 및 데이터 입력 — 실습 (20분)
  6. 이상 상황 신고 방법 (10분)
- 교육 형태: 집합 교육 + 실습
- 수료 기준: 실습 과제 완료

**교육 과정 2: 고급 사용자 교육 (품질관리·공정 엔지니어)**

- 대상: 품질팀 품질관리 담당자, 기술팀 공정 엔지니어
- 교육 시간: 4시간
- 교육 내용:
  1. ML 모델 원리 및 성능 지표 이해 (40분)
  2. 데이터 분석(EDA) 화면 활용 — 실습 (40분)
  3. 모델 현황 화면 — 성능 모니터링 방법 (30분)
  4. DOE 시뮬레이터 활용 — 실험 설계부터 결과 해석까지 — 실습 (60분)
  5. 배합비율 확정 의사결정 프로세스 (30분)
  6. 불량 원인 분석 방법 (40분)
- 교육 형태: 집합 교육 + 실습 + 사례 연구
- 수료 기준: DOE 시뮬레이션 실습 과제 제출

**교육 과정 3: 운영자 교육 (시스템·데이터 관리자)**

- 대상: 스마트팀/IT팀 시스템 관리자, 데이터 관리자
- 교육 시간: 8시간 (2일)
- 교육 내용:
  1. 시스템 아키텍처 이해 (Backend/Frontend/ML 파이프라인) (60분)
  2. 서버 운영 및 배포 절차 (60분)
  3. 학습 데이터 관리 및 전처리 프로세스 — 실습 (90분)
  4. 모델 재학습 실행 및 성능 평가 — 실습 (90분)
  5. 장애 대응 시나리오 실습 (60분)
  6. 백업 및 복구 절차 실습 (60분)
  7. 운영 모니터링 및 보고서 작성 (30분)
- 교육 형태: 기술 교육 + 실습 + 시나리오 훈련
- 수료 기준: 모델 재학습 및 장애 복구 실습 완료
