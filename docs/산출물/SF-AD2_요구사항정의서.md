## 1. 문서 개요

### 1.1 목적

본 문서는 **Formulation ML 시스템** (성분분석 데이터 기반 배합비율 최적화 ML 시스템) 구축을 위해 각 이해관계자가 요구하는 기능적·비기능적 요구사항을 체계적으로 정의하고 문서화하는 데 목적이 있다.

정의된 요구사항은 설계·개발·테스트·검수의 기준으로 활용되며, 이해관계자 간 공통된 기대치를 확립하여 프로젝트 범위 관리의 근거로 사용된다.

### 1.2 적용 범위

- **대상 시스템**: Formulation ML (배합비율 추천, 품질 예측, EDA, DOE 시뮬레이터, 모델 모니터링)
- **기술 범위**: Backend(FastAPI, Python ML), Frontend(Next.js 14, TypeScript)
- **업무 범위**: 납땜 합금(솔더) 배합 공정 — 배합비율 결정, 품질 예측, 실험 계획

### 1.3 요구사항 분류 체계

| 분류 | 코드 접두사 | 설명 |
|------|-------------|------|
| 기능 요구사항 | FR | 시스템이 수행해야 하는 기능 |
| 비기능 요구사항 | NFR | 성능, 가용성, 보안 등 품질 속성 |
| 제약 사항 | CON | 기술·조직·법적 제약 조건 |

### 1.4 우선순위 정의

| 등급 | 기준 |
|------|------|
| 필수 (Must) | 없으면 시스템 핵심 목적 달성 불가. 1차 오픈 필수 포함 |
| 높음 (High) | 주요 사용자 가치 제공. 1차 오픈 포함 권장 |
| 중간 (Medium) | 유용하나 없어도 운영 가능. 2차 오픈 포함 |
| 낮음 (Low) | 편의 기능. 일정·자원 여유 시 포함 |

### 2.2 이해관계자별 요구사항

**STK-01 납땜공정팀 (제조팀)**

현장 기술원과 수석 기술원의 핵심 요구사항은 다음과 같다.

- 공정 온도, 용융 시간, 공급사 정보를 입력하면 즉시 최적 배합비율(SN/AG/CU/PB)을 추천받을 수 있어야 한다.
- 추천 결과의 근거(피처 중요도, 예측 품질 점수)를 이해할 수 있는 형태로 제시해야 한다.
- 비전문가도 사용할 수 있도록 직관적인 UI여야 하며, 별도 교육 없이 30분 내 사용 가능해야 한다.
- 공급사가 변경될 때 배합비율 조정 가이드가 자동으로 제시되어야 한다.

**STK-02 품질팀**

- 배합 전 품질 점수를 예측하여 고위험 조건을 사전에 식별할 수 있어야 한다.
- 75점 미만 불량 예상 시 경고 알림을 받을 수 있어야 한다.
- LOT별 예측값 vs 실측값 비교 리포트를 조회할 수 있어야 한다.
- 성분 분포 및 목표값 대비 편차 추이를 시각화된 형태로 확인할 수 있어야 한다.

**STK-03 IT팀**

- REST API 형태로 외부 시스템 연동이 가능해야 한다.
- 시스템 관리자 기능(사용자 관리, 로그 조회, 모델 업데이트)이 있어야 한다.
- 온프레미스 또는 내부 클라우드 환경에 배포 가능해야 한다.
- API 문서(Swagger/OpenAPI)가 자동 생성되어야 한다.

**STK-04 경영진**

- 월간/주간 공정 품질 KPI(불량률, 평균 품질 점수) 요약 대시보드를 열람할 수 있어야 한다.
- 스마트공장 투자 효과(비용 절감, 불량률 감소) 지표를 시각화하여 보고할 수 있어야 한다.

**STK-05 R&D팀**

- 다양한 DOE 설계(Full Factorial, CCD, Taguchi 등 6종) 기능을 활용하여 실험 조건을 빠르게 탐색할 수 있어야 한다.
- DOE 시뮬레이션 결과를 엑셀(CSV) 형태로 다운로드할 수 있어야 한다.
- 새로운 ML 모델을 학습시키고 성능을 기존 모델과 비교할 수 있어야 한다.

| NFR-004 | 보안 | API 인증 및 인가 | FastAPI 엔드포인트는 JWT 토큰 기반 인증을 적용한다. 역할(Role) 기반 접근 제어로 일반 사용자/관리자 권한을 구분한다. 민감한 API(모델 교체, 데이터 삭제)는 관리자 권한만 허용한다. | 침투 테스트, 인증 우회 시도 차단 확인 |

| NFR-006 | 유지보수성 | 코드 품질 및 문서화 | Python 코드는 PEP 8 스타일 가이드를 준수한다. 주요 함수 및 클래스에 docstring을 작성한다. TypeScript 코드는 ESLint 기준을 통과해야 한다. API 문서는 Swagger/OpenAPI 형식으로 자동 생성된다. | 정적 분석 도구(flake8, ESLint) 경고 0건 |
| NFR-007 | 유지보수성 | 모델 버전 관리 | ML 모델 파일(.joblib)과 전처리기는 버전 정보(학습일, 데이터 크기, 성능 지표)와 함께 저장되어야 한다. 이전 버전으로 롤백이 가능해야 한다. | 모델 메타데이터 파일 존재 확인, 롤백 절차 테스트 |
| NFR-008 | 확장성 | 모델 추가 확장성 | 새로운 ML 모델 알고리즘을 REGISTRY 딕셔너리에 추가하는 방식으로 시스템 재구축 없이 모델을 확장할 수 있어야 한다. 신규 모델 추가 시 코드 변경 범위는 models/train.py의 REGISTRY 딕셔너리 수정으로 제한된다. | 신규 모델 추가 소요 시간 (2시간 이내) |
| NFR-009 | 호환성 | 브라우저 호환성 | 웹 UI는 Chrome 최신 2개 버전, Edge 최신 2개 버전에서 정상 동작해야 한다. Internet Explorer는 지원하지 않는다. | 브라우저 호환성 테스트 체크리스트 |
| NFR-010 | 운영성 | 에러 로깅 및 모니터링 | API 서버는 요청/응답 로그를 구조화된 형식(JSON)으로 기록한다. 5xx 에러 발생 시 로그에 스택 트레이스가 포함되어야 한다. 로그는 최소 30일 보관한다. | 에러 발생 시 로그 확인 테스트 |

## 5. 제약 사항

### 5.1 기술적 제약

| ID | 제약 유형 | 내용 |
|----|-----------|------|
| CON-001 | 기술 스택 | Backend는 Python 3.10 이상, FastAPI 0.100 이상을 사용한다. ML 라이브러리는 scikit-learn, XGBoost, scipy로 제한한다. |
| CON-002 | 기술 스택 | Frontend는 Next.js 14 App Router, TypeScript, Tailwind CSS를 사용한다. UI 컴포넌트 라이브러리는 shadcn/ui 또는 자체 구현을 원칙으로 한다. |
| CON-003 | 포트 | Backend는 port 8000, Frontend는 port 3000으로 운영한다. CORS 설정으로 Frontend에서 Backend API 호출을 허용한다. |
| CON-004 | 데이터 형식 | ML 학습 데이터는 CSV 형식(컬럼: lot_id, sn_pct, ag_pct, cu_pct, pb_pct, other_pct, melt_temp_c, melt_time_min, supplier_id, quality_score, is_defect)을 표준으로 한다. |
| CON-005 | 모델 저장 | 모델 파일은 .joblib 형식으로 저장하며, 모델과 전처리기는 같은 이름으로 쌍 저장한다: {name}.joblib + preprocessors_{name}.joblib |
| CON-006 | 배포 환경 | 아모레퍼시픽 내부 네트워크 환경에 배포한다. 외부 인터넷 접속이 제한될 수 있으므로 모든 종속성은 사내 배포 가능해야 한다. |

### 5.2 업무적 제약

| ID | 제약 유형 | 내용 |
|----|-----------|------|
| CON-007 | 성분 합계 | 배합 추천 결과는 SN + AG + CU + PB + other의 합이 100%가 되어야 한다 (SLSQP 최적화 등호 제약). |
| CON-008 | 목표값 | SN_TARGET=62.0%, AG_TARGET=3.0%, CU_TARGET=0.5%를 피처 엔지니어링 기준값으로 사용한다. 변경 시 engineering.py 상수 수정 및 모델 재학습 필요. |
| CON-009 | 불량 기준 | 품질 점수 75점 미만을 불량으로 정의한다. 이 기준은 관리자 설정 화면에서 변경 가능하나, 변경 시 모델 재학습 권고 사항을 안내한다. |
| CON-010 | 공급사 | 현재 등록된 공급사는 SUP_A, SUP_B, SUP_C 3개사이다. 신규 공급사 추가 시 SUPPLIER_EFFECTS 딕셔너리 업데이트 및 모델 재학습이 필요하다. |

### 5.3 일정·자원 제약

| ID | 제약 유형 | 내용 |
|----|-----------|------|
| CON-011 | 일정 | 스마트공장 보급·확산사업 지원금 정산 기한에 따라 시스템 구축 완료 및 검수는 사업 종료 1개월 전까지 완료해야 한다. |
